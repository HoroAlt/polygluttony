# polygluttony — guide for AI coders

Read this before writing any code. It explains **what we're building, how to think
about the port, and which references are authoritative.** It complements `README.md`
(human-facing stack/setup) — this file is the *working philosophy*.

---

## 1. The mission: a full Python → Rust rewrite

We are rebuilding the Python app at **`../subs-translate/`** as **polygluttony**, a
cross-platform Tauri 2 desktop app:

- **Engine** → Rust, living **inside `src-tauri/src/`** (modules `ass/`, `llm/`,
  `glossary/`, `validation/`, `translation/`, plus `config/`, `models/`, `events.rs`,
  `commands/`). There is **no separate crate and no Cargo workspace** — by design.
- **UI** → React 19 + TypeScript in `src/`, Tailwind v4 + shadcn (maia) + TanStack
  Router/Query. See `README.md` for the full stack.
- The two halves meet only at the **IPC boundary**: Rust `#[tauri::command]`s and
  emitted events, with TypeScript types generated by `ts-rs` into
  `src/types/generated/` (run `bun gen:bindings` after changing a Rust `TS` type).

## 2. Scope: only core logic that serves the UI — no CLI

The Python project ships both a Typer/Rich **CLI** and a PyQt6 **GUI**. **We are
porting only what the GUI/UI needs.**

- **Do not** port CLI concerns: argument parsing, `-y/--yes` confirmation prompts,
  Rich/TUI rendering, `cli/` wiring, the `--only-glossary/--only-translate/...` modes.
- The engine exists **solely to serve the webview** through commands + events. It has
  no other public surface. If a piece of Python logic only existed to support the CLI,
  drop it. CLI progress rendering (Rich) is replaced by **events streamed to the UI**.

## 3. We rewrite — we do not transcribe

**This is not a line-by-line port.** The Python code is the *behavioral* reference, not
a style template, and it is **not necessarily good code**. For each subsystem:

1. Read the Python and understand what it does and why.
2. Then write **idiomatic, well-structured Rust** that achieves the same outcome —
   stronger types, clear module boundaries, real error handling (`thiserror`/`AppError`),
   `tokio` async with bounded concurrency. **Improve** awkward designs; don't faithfully
   reproduce their flaws.
3. Where the **new UI** changes behavior (see §5 — modes removed, verify = issues not a
   score, connections have a real GUI), follow the UI spec, not the old Python flow.

## 4. …but respect the load-bearing details — understand *why* before you change them

The Python encodes **hard-won correctness against real LLM failure modes.** Rewrite the
*code* freely, but preserve the *intent* of these — and if you're about to simplify one
away, first read the cited source and the core-logic doc and articulate why it exists.

| Detail (Python location) | Why it exists — don't discard blindly |
|---|---|
| **Line markers** `<0001:D>` / `<0001:L>`, fuzzy recovery, partial-success fallback (`validation/line_marker.py`, `core/batch_translator.py`) | LLMs merge, drop, and reorder lines. Markers detect *exactly where* it broke and salvage the correct prefix instead of failing the whole batch. |
| **5-signal weighted drift detector** — punctuation, glossary position, sentence type, last line, length ratio; alert below 0.7 (`validation/`, `validation/signals/`) | Structural checks pass but content can still *drift* mid-batch. The weighted signals catch it and pinpoint the retry start. |
| **ASS tag preservation** — strip inline override tags, track by position, reapply (`parsers/`) | Only the dialogue text should be translated; `{\pos}`, `{\an8}`, font tags, styles, and metadata must come back byte-faithful. |
| **Glossary as context** — six categories + world-type detection (`glossary/`) | Keeps names/terms consistent across hundreds of episodes; world type (xianxia/wuxia/historical/modern) tunes extraction and tone. |
| **Token-aware batching + per-connection concurrency** (`core/batch_manager.py`, `core/scope_calculator.py`) | Maximize throughput without exceeding model context or provider rate limits. |
| **Cleanup pass** for residual source text (`core/translator.py`) | LLMs leave fragments untranslated; an iterative re-pass with surrounding context fixes them. |
| **Driver differences** — Anthropic extended thinking, OpenAI chat, OpenAI Responses web search (`llm/`) | Each provider has a distinct request/stream/response shape that the abstraction must accommodate. |

## 5. The UI we build toward: `../polygluttony-docs/`

A **new UI was designed and visually validated for subs-translate but never built.**
polygluttony implements that design. The spec lives in **`../polygluttony-docs/`** and is
authoritative for *what the UI does and how each action binds to the engine.*

- **Information architecture:** a **single window with a left icon rail** swapping
  focused views — **not tabs, not a wizard.** Views: **Project, Glossary, Translate,
  Verify, Connections, Settings, Help**, plus a pre-project **Welcome** screen and the
  **shell** (rail + header + status bar).
- ⚠️ **The current scaffold's top-nav tabs (`src/routes/`, `MainNav`) are a placeholder.**
  Replace them with the icon-rail + full view set from the docs during implementation.
- Each window doc has an **Interactions table** citing the precise Python `file:line` +
  function each control binds to. `03-operations-and-flows.md` defines the 21 logical
  operations (O1–O21), the **progress/log/state event contract** (`core/progress.py`:
  `TranslationProgress`, `TranslationState`), the per-folder state model, and **gating
  rules**. `02-core-logic-reference.md` is the engine's source-of-truth map.

**Reading order**
- UI work: `polygluttony-docs/00-overview.md` → `01-design-system.md` → `windows/00-shell…` → each view.
- Engine work: `polygluttony-docs/02-core-logic-reference.md` → `03-operations-and-flows.md`, cross-referencing `../subs-translate/subs_translator/…` as the behavioral source of truth.

**Key product facts** (don't lose these): glossary is **optional**; old modes are gone
(now just navigation); Connections is a real GUI with provider presets (incl. **Gemini**)
+ a **Test** button (no hand-editing `config.yml`); **Verify = an issue list, never a
score**, runs automatically during translation and is re-runnable; **world type is
auto-detected** instantly via keyword heuristic (no LLM) and is overridable.

> Note: `polygluttony-docs` intentionally makes **no Rust/porting decisions** (crates,
> async strategy, types). Those are ours to make here, guided by §3 and §4.

## Map of this repo

```
src-tauri/src/
  commands/   # the entire IPC surface the webview can call
  events.rs   # progress/log/state events emitted to the UI
  config/     # AppConfig / Connection / Driver (persisted via Tauri store plugin)
  models/     # shared types crossing the IPC boundary (derive ts-rs `TS`)
  error.rs    # AppError (serializes to a string for the webview)
  ass/ llm/ glossary/ validation/ translation/   # the engine (ports of subs_translator/*)
src/
  routes/     # TanStack file-based routes (PLACEHOLDER nav — see §5)
  features/   # per-view UI: glossary, translate, verify, settings (+ project/connections/welcome/help to add)
  components/  lib/  stores/  hooks/
  types/generated/   # ts-rs output — DO NOT edit by hand; `bun gen:bindings`
```

## Before you claim something works

- Frontend: `bun run build` (route gen → tsc → vite). Backend: `cargo build`.
- After changing a Rust `TS` type: `bun gen:bindings`, and check `src/types/generated/`.
- Config is stored via the **Tauri store plugin** (JSON in app-data), **not** the old
  `~/.config/subs-translator/config.yml`.
